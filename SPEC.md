# Q/GDW 10376.2 集中器本地通信模块 — 解析工具规格与开发需求

> **协议依据**：仓库内 `doc/17 用电信息采集系统通信协议 第2部分：集中器本地通信模块接口.pdf` 正文为 **Q/GDW 10376.2—2019**（代替 Q/GDW 1376.2—2013）。  
> **实现主体**：`GDW1376_2/` 下 C 源码编译为 `GDW1376_2.dll` / `GDW1376_2_64.dll`；`py103762` / `pyqt4_103762` 通过 `ctypes` 调用导出符号 `GDW1376_2_parse_foy_py`；`gw13762` 直接链接 `GDW1376_2_parse`。

---

## 1. 目标与非目标

### 1.1 产品目标

- **输入**：十六进制字符串（可含空格/换行）；可选扩展为「多帧粘包自动切分」。
- **输出**：人类可读的中文解析（链路层字段 + 应用层 AFN/FN 数据单元）；错误时返回**稳定错误码**与说明。
- **约束**：解析逻辑与 **10376.2—2019** 一致；对厂商扩展 AFN=F0H 仅做**十六进制转储**或明确标注「非标准」。

### 1.2 非目标（当前阶段可不实现）

- 645/698 应用层完整语义解析（仅透明展示负载即可）。
- 主动发包、会话状态机、与真实串口的绑定（可做独立工具）。

## 2. 系统结构与接口约定

### 2.1 模块划分

| 层级 | 路径/产物 | 职责 |
|------|-----------|------|
| 核心解析 | `GDW1376_2/src/*.c` | 帧校验、结构打印、AFN/FN 分发与字段解析 |
| 公共 API | `GDW1376_2/inc/GDW1376_2.h`、`GDW1376_2.def` | `GDW1376_2_parse`、`GDW1376_2_parse_str`、`GDW1376_2_error`、`GDW1376_2_parse_foy_py` 等 |
| Python 封装 | `py103762/master/trans/translate.py` | 加载 DLL、GBK 解码输出 |
| Qt 界面 | `gw13762/mainwindow.cpp` | 输入十六进制、回调拼接到文本框 |

### 2.2 返回值约定（需统一）

- **成功**：`0`（`ERR_NONE`）。
- **失败**：**负的** `ERR_*`（与 `GDW1376_2_error()` 表一致）。
- **禁止**：混用 `return -1` 与 `ERR_*`；禁止「粘包」场景返回**正数**错误码（见缺陷 P0-02）。

### 2.3 多 FN（DT1 多 bit）支持

- 现状：`lib.c` 中 `get_fn` / `get_clear_fn` 仅取 DT1 **最高**置位 bit，**多 FN 帧会静默丢 FN**。
- 需求：要么**明确不支持**并在结构解析处打印警告 + 返回 `ERR_AFN_FN`（或新增 `ERR_MULTI_FN`），要么实现**循环清除 bit 并依次解析**（与协议组帧规则一致时再选后者）。

## 3. 协议摘要（与实现对齐用）

### 3.1 帧与长度

- 起始 `68H`，2 字节长度 **L**（低字节在前），控制域 **C**，用户数据区 …，校验 **CS**，结束 `16H`。
- **L** = 用户数据长度 **L1** + 6（含起始、L 本身、C、CS、结束等固定部分 —— 以标准文本为准）。
- **CS**：从 **C** 起至 CS 前一字节的算术和（溢出丢弃），实现位于 `get_cs` + `chkfrm_GDW1376_2`。

### 3.2 控制域 C

- D7：**DIR**（0 集中器→模块，1 模块→集中器）
- D6：**PRM**
- D5~D0：通信方式（1 集中式路由载波、2 分布式、3 宽带/HPLC、10 微功率无线、20 以太网等 —— 以 PDF 表为准）

### 3.3 地址域 A

- 当信息域中「通信模块标识」表示**对从节点操作**时出现 **A1/A2(仅下行)/A3**；实现用 `user_data_pos` 定位应用数据起点（见 `GDW1376_2.c` `print_struct`）。

### 3.4 应用数据

- **AFN**（1B）+ **DT1** + **DT2** + 数据单元。
- **Fn**：`Fn = DT2×8 + (DT1 中置 1 的位序号)`；若 DT1 多位为 1，则存在多 FN（见 2.3）。

## 4. 源码缺陷登记（按优先级）

> 说明：下列为基于当前仓库源码的**可复现/可读性**问题；修复时以 **10376.2—2019** PDF 数据单元表为验收依据。

### P0 — 解析错误、越界或错误码破坏契约

| ID | 位置 | 现象 |
|----|------|------|
| P0-01 | `GDW1376_2.c` `print_struct` 上行 R 解析 | `pin[6]` 为单字节，使用 `pin[6] >> 8` 取「电能表通道特征」恒为 0，应取 **高 4 位**等与协议一致的字段拆分。 |
| P0-02 | `chkfrm.c` | 当 `packlen < len` 时返回 **正数** `ERR_CHKFRM_LEN_LONG`，与「错误为负」惯例不一致，且与 `GDW1376_2_error()` 索引方式不统一，易造成调用方误判。 |
| P0-03 | `AFN10.c` `AFN10_FN04` | 「路由完成标志」与「工作标志」位域复用/错位：`BITS(pin[0],0)` 同时承担不同语义；工作标志应使用 **D1** 等协议定义位（与旧 SPEC BUG-04 一致）。 |
| P0-04 | `AFN10.c` `AFN10_FN21` | 起始序号组合字节错误：`pin[2] \| ((int)pin[2]<<8)` 应为 **`pin[2] \| (pin[3]<<8)`**。 |
| P0-05 | `AFN10.c` `AFN10_FN31` | 上行长度校验使用 `pin[2]`，与「本次应答数量 `pin[4]`」不一致，易导致长度误判或越界。 |
| P0-06 | `AFN10.c` `AFN10_FN07` | 长度公式使用 `pin[12]`、循环内混用 `i*12` 与 `i*8`，且 `idformat` 判断误用 `pin[0]`；整体与「变长 ID」结构不符，存在**解析错误与越界风险**。 |
| P0-07 | `AFN10.c` `AFN10_FN40` | 下行 `ID类型` 判断写成 `(pin[0]<3)`，应对 **ID 类型字节**（如 `pin[7]`）判断；当前会错误映射 `idtype[]`。 |
| P0-08 | `AFN10.c` `AFN10_FN111` | 邻居数量循环使用 `pin[2]`（中间字节），应为 **`pin[0]`（多网络节点总数）** 一类字段（与旧 SPEC BUG-07 一致）。 |
| P0-09 | `AFN03.c` `AFN03_FN11` | FN 位图遍历 `BITS(pin[1+i/8], i)` 未取 **bit 序号 `i%8`**，导致支持的 Fn 列表错误。 |
| P0-10 | `AFN14.c` `AFN14_FN01` | 下行 `pin[0]==2` 分支：附属节点数量、地址偏移大量使用 **`pin[3+pin[3]]`** / `pin[3+pin[2]]` 混用及 `for(j=1; j<pin[3+pin[3]]` 等错误，和协议「L + 数据 + n + 地址列表」布局不一致，**必然解析错误**。 |
| P0-11 | `AFN13.c` `AFN13_FN01` | 上行打印报文内容使用 `pin[i + pos + 4]`，而 `pos` 在 **dir==1** 时未赋值（来自下行的附属段长度），上行解析**错位**。 |
| P0-12 | `AFN.c` `print_AFN` | 未识别 AFN 时 `return -1`，应返回 **`-ERR_AFN_FN`**（或专门 `ERR_AFN`），否则 UI/Python 与 `GDW1376_2_error()` 不一致。 |

### P1 — 协议/语义不完整或与 2019 版不一致

| ID | 位置 | 现象 |
|----|------|------|

| P1-04 | `AFN05.c` `AFN05_FN03` | ~~控制字数组/todo~~ **已对齐表 41**（含相位识别）；并校验 **2+L** 总长。 |

### P2 — 工程质量与安全债

| ID | 位置 | 现象 |
|----|------|------|
| P2-01 | 全局 `buf[256]` + 大量 `sprintf` | 长行、长报文有缓冲区溢出风险；应统一 `snprintf` 或封装 `pcb_hex/pcb_fmt`。 |

## 6. 开发需求（分阶段、可验收）

### 阶段 M0 — 基线与安全（约 1 周）

1. 统一错误码：`chkfrm` 粘包分支、未知 AFN、各 AFN `return -1` 全部改为 **`ERR_*` 负值**；更新 `GDW1376_2_error` 文案。  
2. 修复 **P0-01、P0-02、P0-12**。  
3. `main.c` **P2-05**；`GDW1376_2_parse_foy_py` 命名与头文件导出（或保留旧符号做 **deprecated 别名**）。  
4. 增加最小单元测试或脚本：用 `build/测试报文.txt` + 手工构造短帧，断言**返回值**与关键字段子串。

**验收**：短帧/错帧不崩溃；错误码与 `GDW1376_2_error` 一致；测试报文仍能解析。

**M0 已落地（源码）**：`chkfrm.c` 粘包返回 **`-ERR_CHKFRM_LEN_LONG`**；`GDW1376_2.c` 上行 **R[2] 电能表通道特征** 使用 **`pin[6] >> 4`**；`AFN.c` 未知 AFN 返回 **`-ERR_AFN_FN`**；`main.c` `printf("%s", …)`；`inc/GDW1376_2.h` 增加 **`GDW1376_2_parse_foy_py`** 声明；`scripts/smoke_m0.py` 冒烟（优先加载 `GDW1376_2/GDW1376_2_64.dll`）。**未纳入 M0**：各 AFN 文件内大量 `return -1` 仍待分阶段改为 `ERR_*`（见 §4 P0 列表其余项）。

### 阶段 M1 — 路由与查询核心（约 2~3 周）
